## 帖子 6：用龙虾跑通 Boss 直聘的自动化招聘及总结（重点）

| 字段 | 内容 |
|------|------|
| 作者 | 复旦馒头-连续创业再战AI |
| 发布时间 | 2026-04-01，上海（编辑版本） |
| 互动 | 194♥ / 251⭐ / 69评论 |
| 原文链接 | https://www.xiaohongshu.com/explore/69ccceb200000000210056a8 |

**技术方案**

OpenClaw（龙虾）**本地版**，配合飞书（Feishu）API 集成。云端版龙虾经测试无法绕过 Boss 检测。

**实现功能**

1. 按岗位 JD 在 Boss 平台筛选简历
2. 与合适候选人沟通约面
3. 通过飞书创建面试日程
4. 通过飞书为面试官创建待办任务

**遇到的问题（详细记录）**

1. **Token 消耗高、速度慢**：龙虾通过操作本地网页获取候选人信息，效率低，需要探索更高效的信息提取策略
2. **频繁触发 Boss 风控**：容易跳出人工验证（CAPTCHA），阻断自动化流程
3. **AI 选人标准漂移**：除岗位 JD 外，需配置公司统一招聘标准（固定规则记忆），防止每次 AI 判断标准不一致
4. **需设计人工介入规则**：区分哪些候选人 AI 全程自动处理、哪些需审批、哪些直接淘汰、哪些提交人工复核

**作者建议与注意事项**

- 正在撰写 Skill 供社区使用，但 Boss 风控在持续升级，撰写过程中原本有效的浏览器工具已失效，需重新跑通
- 龙虾和 Agent 的"智商"导致存在不少失控行为，Token 浪费问题需逐一约束

**评论区补充（高价值内容）**

- `openclaw能访问boss吗？为什么我用直接就被检测到了，然后登录状态就被清除了`（广东用户）——作者回复：我用的是**本地版**，云端版换虚拟浏览器登录是大概率不行的
- `先用影刀在boss操作，返给openclaw对话内容或者简历，这样不会触发封控`（重庆用户）——提出**影刀 RPA 作桥接层**的替代方案
- `如果是一直要做招聘的小伙伴，劝你们用工具跑慎重。Boss的风控是全网最严格的，Boss直聘的算法是记录个人账号权重的，这个权重不是你换了公司就能解决的。一旦被标记限流，真的哭都来不及。测试下来，**一次 30 个是极限了**。`（河南用户）
- `我也在做这块，目前因为boss的反扒比较强。主要是在猎聘平台。`（安徽用户）——已转移阵地

**社区对可行性的判断**

- 高互动量（194♥/251⭐）说明话题热度高，但评论内容充满技术卡点描述
- 本地 OpenClaw 可以跑通，但稳定性差，Boss 风控在主动进化；云端版基本无效
- 单次批量上限约 30 个，否则触发限流且账号权重降权后无法恢复

## 反复出现的关键信息点

**Boss 直聘反爬机制**

- Boss 风控被多篇帖子（帖子 1、5、6）确认为国内招聘平台中最严格
- 账号权重模型：一旦被标记限流，换公司也无法解除，需换新账号
- 单次批量操作上限约 30 个（来自有实测经验的评论用户）
- 云端自动化基本无效（登录态立即被清除）；本地方案可用但仍频繁触发 CAPTCHA

**本地 vs 云端**

- 多篇帖子均指向"本地浏览器方案优于云端虚拟浏览器方案"
- 飞书云端部署的 OpenClaw 被明确指出对 Boss 无效

**替代平台**

- 多名用户表示在 Boss 受阻后转向猎聘（帖子 1 评论、帖子 6 评论）
- 51job、猎聘、智联被认为反爬较弱，适合自动化

**法律与合规担忧**

- 帖子 1 评论区有用户提出"扒简历是否违法"的问题
- 帖子 7 评论区有用户指出"AI 初筛会直接淘汰 GAP 一两年的人，细思极恐"

## 二、技术方案分类汇总

### 方案1：Selenium 方案

**代表项目/文章**

- 掘金《Boss直聘自动打招呼脚本 Python3 + Selenium》（2023年，zdjzce923/automate-boss）
- 知乎《使用selenium实现BOSS爬虫自动投递》

**技术路径**

- 使用 Selenium WebDriver 操控 Chrome/Firefox
- 移动端模式（iPhone XR）比PC端更容易操作
- 通过 user-data-dir 保持登录态（指定Chrome配置文件路径）
- remote-debugging-port 固定端口避免每次新建实例

**遇到的问题/限制**

- 滑块验证码：登录时触发概率性滑块验证，实现过程复杂
- 页面元素频繁变动，XPath维护成本高
- Selenium 被检测风险高，Boss直聘可识别 Selenium 特征

**避坑建议**

- 优先使用移动端UA，可降低被检测概率
- 固定 user-data-dir 路径对保持登录态至关重要
- 不自动关闭浏览器（detach: True），下次复用

**当前可行性**：中低。Selenium特征明显，2024-2025年社区反馈被检测概率显著上升。

---

### 方案2：Playwright 方案

**代表项目/文章**

- 《Python爬虫实战：Boss直聘职位数据采集实战 - Playwright + 结构化解析完整方案》（2026年2月）
- 《使用Playwright和curl_cffi爬取BOSS直聘职位信息》（知乎+csdn，2025年）
- ufownl/auto-zhipin（GitHub，Playwright + 简历智能筛选 + 个性化打招呼）

**技术路径**

- Playwright 替代 Selenium（更轻量、特征更少）
- curl_cffi 配合 Playwright 更新 Cookie，突破反爬
- 监听网络接口直接获取 JSON 数据（绕过页面渲染）
- 无头模式（headless）运行降低被检测风险
- DrissionPage（基于Playwright封装）也是常用框架

**遇到的问题/限制**

- 2025年Boss直聘做了三个关键改动：
  1. 反爬检测：打开F12开发者工具即触发无限循环或闪退
  2. 分页机制取消：以前 page=N 参数翻页方式失效，改为滚动加载
  3. 薪资数据加密：列表页薪资数字字体加密变成乱码，但详情页薪资仍是明文
- zp_stoken 参数必须携带，否则请求被拦截
- 效率较低：约300个岗位详情需要20分钟

**避坑建议**

- 用 DrissionPage 的禁用JavaScript模式绕过反爬检测
- 直接解析详情页薪资（明文），绕过列表页加密
- 滚动加载适配新的无分页模式
- 自动化推荐用 Playwright 的 chromium 配合 --disable-blink-features=Automation

**当前可行性**：中高。Playwright 仍是主流有效方案，但需要适配最新的反爬机制。

### 方案3：油猴脚本 / Chrome 扩展方案

**代表项目/文章**

- 掘金《BOSS直聘自动打招呼油猴脚本》（2024年，作者：迷途小码农）

- V2EX《基于 AI 大模型的 Boss 直聘打招呼文案生成 Agent》（2025年）
- boss-helper（Chrome Web Store，6000+ 用户，活跃维护）

**技术路径**

- 油猴脚本/Tampermonkey 注入 JS 到 Boss直聘页面
- 在页面内直接操作 DOM，无需浏览器自动化工具
- 配合微信推送（pushplus）通知执行结果
- GPT API 生成个性化打招呼文案

**功能实现**

- 自动翻页、滚动加载更多职位
- 按薪资/HR在线状态过滤
- 自动进入详情页打招呼，自动关闭
- 每日沟通上限100次的平台限制
- 列表页UI注入 + 控制面板

**遇到的问题/限制**

- 封号风险：V2EX 2025年4月帖子明确提到"boss最近一直弹窗，禁止自己外挂AI插件，否则可能封号"
- JS脚本注入方式对BOSS直聘版本更新敏感，BOSS改版后脚本容易失效
- Boss直聘会检测注入到页面的第三方脚本特征

**避坑建议**

- 打招呼间隔建议5-10秒随机，不要太频繁
- 每天不要超过70-80次（平台总上限约100次，留余量）
- 优先使用平台原生的打招呼功能，脚本只做自动化点击而非自定义消息

**当前可行性**：中等。油猴脚本方案用户基数大，但BOSS在2025年明显加强了对第三方插件的检测和封号力度。

### 方案4：API逆向方案

**代表项目/文章**

- 掘金《boss直聘zp_stoken逆向分析源码放送》（2023年）
- 博客园《boss直聘__zp_stoken__逆向》（2024年）
- 看雪论坛《boss直聘__zp_stoken__的控制平坦流纯算逆向思路》
- 阿里云《逆向破解boss直聘__zp_stoken__参数的JS混淆与AST插桩》
- gitee《jsrpc--boss直聘：利用jsrpc远程方法调用boos的__zp_stoken__生成》
- CSDN《揭秘Boss直聘-zp-stoken加密算法与实践》

**技术路径**

- zp_stoken 是 Boss直聘的核心安全参数，每次请求必须携带
- 通过浏览器JS调试定位加密逻辑，或用JSRPC直接调用浏览器内生成的token
- JS文件每天0-1点之间变动一次（文件名和内容都会变）
- 核心思路：补环境方案 + Hook 方案

**遇到的问题/限制**

- zp_stoken 每天动态变化：逆向出的代码第二天就可能失效，需要持续维护
- 混淆程度高：控制平坦化 + 大量混淆，逆向成本极高
- 即便拿到zp_stoken：还需匹配正确的浏览器指纹、IP信誉、请求行为模式
- 登录态失效快：Cookie和zp_stoken都有时效性

**作者提到的关键点**

- 上游 mcp-bosszp 架构存在缺陷：登录用Playwright，但调API用requests，导致zp_stoken与请求不匹配被检测
- DrissionPage 方案可绕过zp_stoken解析（通过浏览器环境自动处理）

**当前可行性**：低。zp_stoken 逆向需要持续维护，2025年后BOSS又增加了浏览器指纹和行为分析双重检测，单纯拿到token已不足够。

### 方案5：DrissionPage 方案

**代表项目/文章**

- 技术栈网《通过DrissionPage爬取boss直聘，绕过__zp_stoken__解析》（2024年）
- CSDN《使用 DrissionPage 实现 Boss 直聘招聘信息爬虫》（2024年）
- Gitee《Boos: DrissionPage自动化爬取Boos直聘》

**技术路径**

- DrissionPage 是国产框架，封装了 Playwright 和请求控制
- 同时具备浏览器自动化 + 网络监听能力
- 可直接监听接口响应获取JSON数据，无需解析页面元素
- 支持 cookie 自动管理

**遇到的问题/限制**

- 需要处理BOSS直聘的反爬检测（开发者工具打开闪退）
- 效率一般（1500个详情页采集耗时较长）
- DrissionPage 本身也有被检测的风险

**核心优势**

- 绕过zp_stoken解析（通过浏览器环境自动处理，不需逆向）
- 比纯Selenium更稳定，比纯requests更高效

**当前可行性**：中高。DrissionPage 是2024-2025年较新且有效的方案，社区反馈稳定。

## 四、关键发现

### 封号/检测方面

1. **多层检测机制升级**：BOSS直聘的反爬机制已从单一参数检测升级为多层检测：
   - zp_stoken（JS动态令牌）
   - 浏览器指纹
   - 请求行为模式（频率、间隔、轨迹）
   - IP信誉

2. **2024-2025年是分水岭**：多个文章提到2024年后BOSS显著加强检测，老方案大规模失效。

3. **封号触发场景**：
   - 打招呼频率过高（1000ms一次即触发）
   - 使用第三方JS注入脚本（2025年4月后BOSS弹窗警告）
   - 打开F12开发者工具即闪退（反调试机制）
   - 短时间内大量请求同一接口

4. **封号后恢复**：多个帖子反映申诉无效，账号被封后基本无法解封。

### 登录态保持方面

1. **Cookie + zp_stoken 双重依赖**：仅有Cookie不够，必须携带有效的zp_stoken
2. **每天需要重新登录/刷新**：zp_stoken 有时效性，最长一天有效
3. **二维码登录仍是主流**：自动化登录的最大难题是滑块验证码
4. **多账号管理**：每个Boss账号需要独立的浏览器配置文件（user-data-dir）

### 技术趋势

1. Playwright 正在替代 Selenium 成为主流浏览器自动化方案
2. DrissionPage（国产框架）在2024-2025年获得较多关注
3. AI能力（GPT/Claude）开始与自动化工具结合（生成打招呼文案、简历分析）
4. 油猴脚本方案用户基数大但风险上升

## 五、风险警告汇总

| 风险类型 | 具体表现 | 触发场景 |
|----------|----------|----------|
| **封号/账号风险** | 账号被永久封禁，无法申诉 | 频率过高、JS注入、第三方插件检测 |
| **功能限制** | 每日打招呼上限约100次 | 接近上限后无法继续操作 |
| **验证码拦截** | 滑块/点选验证码阻断自动化流程 | 登录时、高频操作时 |
| **登录态失效** | Cookie/zp_stoken过期 | 每天0-1点更新后需要重新登录 |
| **页面结构变化** | 元素定位失效，脚本整体崩溃 | Boss直聘每次改版 |
| **zp_stoken失效** | 携带过期/错误token的请求被拒绝 | token每天变一次 |
| **IP限制** | 请求过快时被限流或封IP | 批量请求、高频操作 |
| **开发者工具检测** | 打开F12页面闪退或无限循环 | 调试时 |
