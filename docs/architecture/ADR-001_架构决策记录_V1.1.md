# ADR-001 架构决策记录

> **项目：** 笼中仙 AI 招聘产品 · Phase 1 线上初筛
> **文档性质：** 阶段B技术路径验证产出——记录"怎么做、为什么这样做"
> **上游文档：** 《Phase 1 线上初筛 V1 需求说明 V1.1》
> **日期：** 2026年4月
> **版本：** V1.1（审查修订版——补充流程编排、操作路径、去重、岗位画像管理、前置依赖）
> **决策人：** LivingYield

---

## 项目目标

给笼中仙门店店长用的Boss直聘线上初筛助手：系统自动筛选候选人、通过企业微信推送推荐结果，并对高匹配候选人发起首轮接触。第一版目标是AI推荐准确率≥80%、店长筛简历时间减少70%。

---

## 最终决策

| 维度 | 选型 |
|------|------|
| 浏览器自动化 | Playwright（备选 DrissionPage） |
| LLM | DeepSeek V3（后续按能力择优切换） |
| 规则引擎 | 硬编码规则层（Python）+ LLM软性评估层 |
| 推送通道 | 企业微信应用消息 |
| 部署方式 | V1 单台 Linux 服务器集中部署（WSL 开发） |
| 数据存储 | PostgreSQL（候选人数据 + AI判断快照） |
| 后端语言 | Python |

## 交付项逐项决策

### C1：Boss直聘简历获取

**决策：** Playwright + storageState 多 Profile 按需执行

| 决策项 | 结论 |
|--------|------|
| 自动化框架 | Playwright，跑不通再切 DrissionPage |
| 部署 | V1 单台服务器集中部署 |
| 触发方式 | 店长通过企业微信发消息指令按需触发（非定时） |
| 数据源 | 详情面板——优先监听 `/wapi/` 接口拿 JSON，DOM 解析兜底 |
| 登录态管理 | 店长在手机端完成人脸验证 → Playwright storageState 导出 Cookie → 7天自动运行 → 到期通过企业微信推送续期通知 |
| 多账号隔离 | 每个门店一个独立的 browser context + storageState 文件 |

**选择原因：**

- Playwright 的 storageState 天然支持多账号 Cookie 隔离，与近100家独立账号的架构匹配
- 网络请求监听能力可直接拿 `/wapi/` 接口的 JSON 数据，避免穿透三层 iframe 做 DOM 解析
- "按需触发"而非定时运行，大幅降低被检测风险和资源消耗——任意时刻可能只有3-5个账号在执行
- 近100个账号用不同营业执照注册，在平台侧完全独立，同 IP 下多个独立企业分别操作不构成异常模式

**未采用方案：**

- Chrome 扩展方案：门店没有电脑，直接排除
- 多台机器分散部署：V1阶段3-10家试用门店，单台服务器已够用；同 IP 风险在"按需触发+独立账号"前提下可接受
- IP 代理方案：V1不做。增加成本和复杂度，先裸跑观察，遇到问题再加

**待验证项：**

- Playwright storageState 导出的 Cookie 能否在7天内稳定复用（不阻塞主线开发，穿插测试）

**候选人去重策略：**

店长可能多次发"筛选候选人"指令，Boss直聘推荐列表每次刷新内容可能部分重叠。去重规则：

- C1 监听 `/wapi/` 接口拿到的 JSON 中提取候选人唯一标识（encryptGeekId 或等效字段）
- 候选人原始信息表以该标识为唯一键
- 入库时判断：已存在 → 跳过（不重复打分、不重复推送）；不存在 → 新增入库并进入 C2 打分流程
- 若候选人信息有更新（如新增工作经历），V1 暂不处理更新，后续按需补充

### C2：AI匹配打分 + 风险识别

**决策：** 规则引擎 + LLM 混合方案

| 决策项 | 结论 |
|--------|------|
| 架构 | 两层——硬规则层前置过滤 + LLM 层软性评估 |
| 硬规则范围 | 中间版本：红线过滤、年龄范围、学历下限、薪资区间匹配、行业关键词粗筛 |
| LLM 职责 | 稳定性分析、岗位特殊要求识别、工作描述语义理解、综合匹配度评估 |
| LLM 选型 | DeepSeek V3 先行，后续按能力择优切换 |
| 输出格式 | 三档判断（推荐沟通 / 可以看看 / 不建议）+ 风险标注 + 理由 |
| 合并逻辑 | 硬规则红线触发 → 直接"不建议"，跳过 LLM；硬规则通过 → 送 LLM 做软性评估 |

**闭环预留（关键设计约束）：**

每次打分必须存完整判断快照，数据结构包含：

- 候选人原始信息快照（打分时刻版本）
- 硬规则层逐项判断结果（每条规则是否触发、具体数值）
- LLM 层原始输出（完整 JSON：评分、理由、风险标注）
- 使用的岗位画像版本标识
- 最终合并结论
- 时间戳

V1阶段只存不分析。Phase 3 做反馈闭环时，这些数据是"AI判断快照 ↔ 入职后表现"关联分析的基础。

**岗位画像规则管理：**

笼中仙已有 10 份岗位画像文档（doc/ppt/pdf/excel）。C2 编码前需完成前置任务——从这些文档中提取字段级规则，产出标准化配置文件：

- 产出格式：每个岗位类型一个 YAML/JSON 配置，包含硬规则阈值（年龄范围、学历下限、薪资区间、行业关键词白名单/黑名单、红线条件）和 LLM prompt 模板（软性评估维度、岗位特殊要求描述）
- 版本管理：配置文件纳入代码仓库版本控制，每次修改留版本号。判断快照中记录使用的配置版本标识
- 维护方式：V1 阶段由开发者手动维护；后续可开放给 HR 通过简单界面编辑
- 前置时间：在 C1 开发期间同步完成规则提取，不阻塞 C2 编码启动

**选择原因：**

- 混合方案让各层做各自擅长的事：红线规则100%可靠（代码判断不会翻车）、LLM擅长理解非结构化信息（"在某火锅店干过两年"→ 餐饮经验）
- 基层餐饮简历非标准化程度高，纯规则覆盖不了边界情况；纯LLM在红线判断上偶尔翻车会损害店长信任
- 硬规则前置过滤约30-40%明确不合适的候选人，减少LLM调用量，控制成本
- DeepSeek V3 价格最低（输入≈0.1元/千token）、中文能力极强、明确承诺数据不用于训练，满足数据安全约束

**未采用方案：**

- 纯 LLM 方案（方案A）：开发最快但红线规则偶尔翻车，评分一致性差，全量调LLM成本更高
- 纯规则引擎（方案C）：零成本零延迟但无法理解非结构化信息，基层简历信息不完整时容错能力差，达不到80%准确率目标

### C3：筛选报告推送

**决策：** 企业微信应用消息 + 中等信息密度

| 决策项 | 结论 |
|--------|------|
| 推送通道 | 企业微信应用消息（笼中仙已在用企业微信） |
| 信息密度 | 中等版——"推荐沟通"候选人展开摘要（脱敏姓名、核心经验、风险标注）；"可以看看"和"不建议"只显示数字不展开 |
| 架构设计 | 推送模块做通道抽象层，消息内容与格式统一生成，发送通道可切换（V1企业微信，预留飞书接口） |

**选择原因：**

- 笼中仙店长日常使用企业微信，零额外安装成本
- 企业微信应用消息免费、支持富文本卡片、可直接推送给个人用户
- 中等信息密度：店长碎片时间看手机，不点开链接就能看到推荐人关键信息，做出"要不要打招呼"的判断；"不建议"的人无需展开——AI已过滤，不浪费店长注意力

**未采用方案：**

- 飞书机器人：零成本接入但笼中仙不用飞书，需要店长额外安装App
- 微信公众号模板消息：无法主动推送、限制多，不适合按需触发场景
- 极简版信息密度：需额外点击看详情，多一步操作多一次流失
- 详细版信息密度：信息过载，碎片时间看不完

### C4：首轮接触自动化

**决策：** 固定岗位模板 + 店长回复指令确认

| 决策项 | 结论 |
|--------|------|
| 招呼消息生成 | 固定模板——每个岗位类型预设一条招呼语模板 |
| 店长确认方式 | 回复指令——店长看完报告后在企业微信回复指令确认发送对象（如"发1、3"或"全发"） |
| 技术实现 | 复用 C1 的 Playwright + storageState，在已有浏览器自动化基础上增加"定位对话窗口 + 点击打招呼 + 输入模板消息 + 发送"操作 |
| 配额管理 | 系统记录每个账号每日已消耗的主动沟通次数，畅聊版上限50次/天 |

**具体操作路径：**

C4 与 C1 不在同一个 Playwright session 中执行（中间可能隔几小时），需要独立定位候选人。推荐列表是无限滚动、无搜索功能，靠滚动定位特定候选人不可靠。实现策略：

- C1 阶段在监听 `/wapi/` 接口拿 JSON 时，同步记录每个候选人的唯一标识（encryptGeekId）和详情页 URL 路径
- C4 阶段优先尝试：用 Playwright 直接导航到候选人详情页 URL → 点击详情面板的"打招呼"按钮 → 输入模板消息 → 发送
- 如果详情页 URL 方式失效（Boss改版等），降级为：导航到推荐列表页 → 滚动查找目标候选人卡片 → 点击卡片上的"打招呼"按钮
- 待验证：是否存在直接的打招呼 API 接口（如 `/wapi/zpgeek/friend/add.json`），如果有则优先用接口调用，更稳定

**选择原因：**

- 固定模板：V1阶段招呼语的核心目的是"在Boss直聘消息列表里占位"，让店长打开App直接看到这些人；候选人回复率主要取决于岗位吸引力，招呼语个性化影响有限。最小成本验证，如回复率不理想再升级
- 回复指令：店长发"发1、3"比记住交互按钮的含义更直觉，且不依赖企业微信消息卡片是否支持按钮回调，实现确定性更高
- 复用C1技术栈：C4本质是C1的操作扩展，不引入新技术

**未采用方案：**

- LLM生成个性化招呼语（方案B）：多一次LLM调用，V1阶段性价比不高，后续可升级
- 模板+变量填充（方案C）：比固定模板好一点但对信息缺失候选人需要fallback，增加了边界情况处理，V1先用最简方案
- 消息卡片按钮确认（方案A）：体验更好但依赖企业微信消息卡片交互能力，实现不确定性高
- 默认全发+排除（方案C）：违反"AI是副驾"原则，未经明确确认不应自动执行操作

### E1：输入法辅助

**决策：** V1 不做

| 决策项 | 结论 |
|--------|------|
| 是否纳入V1 | 否，作为后续优化项 |
| 备选路径 | A. 开源输入法改造（上下文获取有障碍）；C. 企业微信AI对话（体验朴素但成本最低）；D. 消息代理链路（体验最好但需持续监听新消息，改变C1架构） |
| 决策时机 | V1上线后根据店长实际沟通频率和需求数据再定路径 |

**不做原因：**

- 三条路径都有各自的技术障碍或体验问题，投入产出比不明确
- C1-C4已覆盖核心痛点（翻70%无效简历），E1是锦上添花
- 缺少店长真实使用数据来判断哪条路径值得投入，V1先跑核心链路积累数据

### E2：沟通信息汇总分析

**决策：** 纳入V1开发计划，排最后

| 决策项 | 结论 |
|--------|------|
| 是否纳入V1 | 是，但优先级最低（C1→C2→C3→C4→E2），时间不够可砍 |
| 触发方式 | 店长在企业微信发指令触发（如"分析候选人张三"） |
| 数据获取 | Playwright 从 Boss 直聘沟通 Tab 抓取聊天记录，优先尝试 `/wapi/zpitem/web/chat/message/list/box` 接口，接口不返回完整内容则 DOM 解析兜底 |
| 分析能力 | 聊天记录 + C2 已有简历数据 → DeepSeek V3 生成结构化汇总（关键信息提取、风险点、亮点、是否建议约面试）→ 企业微信推送 |

**选择原因：**

- 技术上是C1能力的自然延伸，不需要新技术栈
- 触发方式复用"店长发指令"模式，与C1-C4交互范式一致
- 对店长有实际价值：聊完几个人后，AI帮忙汇总谁值得约面试

## 系统架构概览

### 触发模式与流程编排

系统有两种触发模式，不是独立任务并行，而是明确的编排链路：

**模式一：筛选链路（一条指令串联 C1→C2→C3）**

```
店长发"筛选候选人"
    → C1：Playwright 读取推荐列表，新候选人入库（去重）
    → C2：对新入库候选人逐个执行硬规则过滤 + LLM评估
    → C3：汇总所有评分结果，生成筛选报告，企业微信推送给店长
    → 等待店长回复
```

**模式二：打招呼（C4 由店长回复报告后单独触发）**

```
店长回复"发1、3"或"全发"
    → C4：解析指令，定位对应候选人，Playwright 执行打招呼操作
    → 执行结果推送给店长（成功/失败/配额不足）
```

**模式三：沟通汇总（E2 独立触发）**

```
店长发"分析候选人张三"
    → E2：Playwright 抓取聊天记录 + 关联已有简历数据 → LLM汇总 → 推送
```

### 架构图

```
店长手机（企业微信）
    │
    │  ① 发送指令
    ▼
企业微信应用后端（Python）
    │
    │  ② 识别店长身份（企业微信userid → 门店 → Boss账号）
    │  ③ 解析指令类型 → 调度对应流程
    ▼
任务调度层
    │
    ├─→ 筛选链路：C1（获取）→ C2（打分）→ C3（推送报告）
    │
    ├─→ 打招呼：C4（Playwright 执行打招呼）
    │
    └─→ 沟通汇总：E2（抓取聊天 → LLM分析 → 推送）
```

### 账号映射

店长发指令时，系统需要知道"这个店长对应哪个 Boss 账号"。映射关系：

```
企业微信 userid → 门店ID → Boss账号ID → storageState文件路径
```

V1 阶段手动配置（3-10 家门店），存储在账号配置表中。

### 数据流

```
Boss直聘（推荐牛人Tab）
    │
    │  C1: Playwright 监听 /wapi/ 接口
    ▼
PostgreSQL
    ├── 候选人原始信息表（唯一键：encryptGeekId，含详情页URL路径）
    ├── AI判断快照表（硬规则结果 + LLM输出 + 岗位画像配置版本）
    ├── 店长决策记录表（采纳/不采纳，为Phase 3预留）
    ├── 账号配置表（企业微信userid ↔ 门店 ↔ Boss账号 ↔ storageState路径）
    ├── 岗位画像配置（YAML/JSON，纳入代码仓库版本控制）
    └── 操作日志表（配额消耗、任务执行记录）
```

## 当前主要风险

| 风险 | 等级 | 应对 |
|------|------|------|
| Boss直聘反自动化升级导致封号或限流 | 高 | "按需触发+低频操作+独立账号"降低概率；社区报告显示低频使用相对安全；V1先跑3-10家观察；触发限流时做指数退避（10s→20s→40s→60s） |
| storageState Cookie 7天有效期未经实测确认 | 中 | 穿插进行Playwright登录态持久化验证，不阻塞主线 |
| Boss直聘前端改版导致接口/DOM结构变化 | 中 | 双重获取策略（接口优先+DOM兜底）降低单点故障；改版后需人工适配 |
| DeepSeek V3 评估质量不达80%准确率 | 中 | Prompt迭代优化；不达标可切换GLM-4或Qwen-Max；硬规则层兜底红线判断 |
| 企业微信应用消息接入需管理员审核 | 中 | 整个系统的入口（店长发指令）依赖企业微信，必须作为与 C1 并行的 Day 1 前置任务启动。需笼中仙企业微信管理员配合创建内部应用并授权消息回调 URL。如果审核流程超过 1 周，C3 开发将被阻塞——备选方案：先用命令行/测试脚本模拟指令输入，企业微信接入后替换入口层 |
| 近100个账号的登录态批量维护 | 低（V1阶段） | V1仅3-10家试用门店，手动维护可接受；规模化时需建设续期提醒和自动化刷新机制 |

## 开发优先级

### 主线（串行）

```
C1（简历获取）→ C2（AI匹配打分）→ C3（报告推送）→ C4（首轮接触）→ E2（沟通汇总）
```

C1是所有后续交付项的基础。C2依赖C1的数据输出。C3依赖C2的评分结果。C4依赖C1的浏览器自动化能力+C3的店长确认指令。E2依赖C1的能力扩展，排最后。

### 并行任务（与 C1 同步启动，不阻塞主线）

| 并行任务 | 说明 | 阻塞谁 |
|----------|------|--------|

## 阶段B完成标准自检

> **操作指南阶段B完成标准：你能清楚回答——前端为什么用这个、后端为什么用这个、怎么部署、当前最大的风险是什么。**

- **为什么用 Playwright：** storageState 天然支持多账号隔离，网络监听能力可直接拿 JSON 数据避免穿透 iframe，社区 Boss 直聘实战案例最多
- **为什么用规则引擎+LLM混合：** 红线规则需要100%可靠（代码判断不翻车），软性评估需要语义理解（LLM擅长），两层各发挥所长
- **为什么用企业微信：** 笼中仙已在用，店长零安装成本，应用消息免费且支持富文本
- **为什么用 DeepSeek V3：** 价格最低、中文能力顶级、明确承诺数据不训练，满足数据安全约束
- **怎么部署：** V1单台Linux服务器，Python后端 + PostgreSQL + Playwright
- **当前最大的风险：** Boss直聘反自动化检测（应对：低频按需+独立账号+指数退避+备选DrissionPage）
