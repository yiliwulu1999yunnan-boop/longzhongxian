## 一句话定义

**给笼中仙门店店长用的Boss直聘线上初筛助手：帮助店长自动筛选候选人、快速看到推荐结果，并对高匹配候选人发起首轮接触，让店长不用再翻70%的无效简历。**

---

## 1. 用户角色

| 角色 | 说明 |
|------|------|
| **店长（第一用户）** | 笼中仙门店一线管理者，兼任招聘负责人。凌晨3-5点到岗，全天忙碌，主要用手机，没时间坐电脑前。招聘不是主业，是"不得不做的事"。 |
| **HR（观察者）** | V1阶段不作为日常操作用户。仅在试用复盘时查看筛选结果、店长采纳情况和效果数据，用于评估系统价值。 |

## 2. 使用场景

### 主场景：早间筛选 + 碎片时间跟进

店长张三管理昆明广场店，店里缺一个面点师。他在Boss直聘上发了岗位，每天收到十几份简历，但大部分是销售、质检等完全不相关的人。

**使用前（现状）：**
张三每天抽空打开Boss直聘App，一份一份翻简历，看到不合适的划走，看到可能合适的点进去细看，再决定要不要打招呼。100份里大概30份值得看，每天花30-60分钟在这件事上，而他凌晨3点就到店备料了。

**使用后（Phase 1 核心目标状态）：**

1. **系统自动筛选：** 系统定时读取张三Boss直聘账号里的新简历，按笼中仙面点师岗位画像自动匹配打分，标记为"推荐沟通""可以看看""不建议"三档，并对风险点（频繁跳槽、行业不匹配等）做标注。

2. **手机收到筛选报告：** 张三在微信/飞书上收到一条推送："今天有12个新简历，3个推荐沟通，4个可以看看，5个不建议。"报告里列出每个推荐候选人的关键信息摘要（经验、风险点、亮点），张三三秒就能看明白。

3. **确认首轮接触：** 张三看了报告，勾选3个推荐的人，确认发起接触。系统在Boss直聘上向这3个人发送一条个性化的招呼消息。

4. **打开Boss直聘接着聊：** 张三打开Boss直聘App，消息列表里已经有这3个人的对话框（因为刚打过招呼）。张三直接点进去，继续跟候选人聊。

**使用后（Phase 1 扩展体验，不属于核心过门槛范围）：**

5. **AI辅助聊天：** 张三跟候选人聊的时候，AI通过输入法提供回复建议，帮他更高效地沟通。

6. **沟通结束后看汇总：** 张三跟几个人聊完了，系统基于简历+聊天记录生成候选人汇总分析，帮他决定约谁来面试。

## 3. 核心问题

**店长每天花大量时间在Boss直聘上翻看不合适的简历，70%是浪费。**

Boss直聘推送的简历中，大部分跟笼中仙岗位不匹配（行业不对、条件不满足），店长只能一份一份人工筛选。筛完之后还要手动逐个打招呼、逐个聊，整个过程低效且重复。

（来源：2026.3.23 HR+店长调研会确认的痛点1和痛点2）

## 4. 第一版范围

### 4.1 核心交付（V1必须有）

| # | 功能 | 说明 | 对应蓝图能力 |
|---|------|------|-------------|
| C1 | Boss直聘简历获取 | 系统读取店长Boss直聘账号中的新候选人简历信息。获取方式（自动化操作/API/半人工）在技术验证阶段确定。 | 对Boss直聘上的候选人简历进行岗位匹配判断 |
| C2 | AI匹配打分 + 风险识别 | 基于笼中仙岗位画像，对每份简历进行匹配度评分，输出"推荐沟通/可以看看/不建议"三档判断。同时识别风险信号：频繁跳槽、行业完全不匹配、关键条件不满足（如面点师不能早起）、触发黑名单规则（一年内离职3次）。 | 对候选人进行风险识别 |
| C3 | 筛选报告推送 | 将筛选结果通过微信/飞书推送给对应店长。报告包含：候选人总数、各档人数、每位推荐候选人的关键信息摘要（经验、亮点、风险点）。店长在手机上三秒看懂。 | 将筛选结果以低负担的方式推送给对应店长 |
| C4 | 首轮接触自动化 | 店长确认后，系统支持对选定候选人发起首轮招呼动作。这是一次性动作，不是多轮自动对话。产品目的是将筛选结果"标记"到Boss直聘消息列表中，让店长打开App就能看到哪些人是已筛选的，直接点进去聊。首轮接触自动化是能力目标，具体实现方式（自动发送/半自动确认发送/其他等效方案）在技术验证阶段确定。 | 提供邀约辅助 |

### 4.2 扩展交付（Phase 1探索范围，非过门槛项）

> 扩展交付属于 Phase 1 探索范围，但不作为 Phase 1 是否过门槛的判断前提。是否纳入当期开发，由技术验证结果与资源评估共同决定。

| # | 功能 | 说明 | 纳入条件 |
|---|------|------|---------|
| E1 | 输入法辅助 | 店长在Boss直聘中与候选人对话时，AI通过输入法提供回复建议（基于候选人简历和对话上下文），类似Lovekey键盘的交互模式。 | 输入法方案技术验证通过 |
| E2 | 沟通信息汇总分析 | 店长与候选人沟通结束后，系统基于简历+聊天记录生成候选人汇总分析（关键信息提取、风险点、亮点、是否建议约面试）。 | 聊天记录获取方案技术验证通过 |

### 4.3 AI筛选依据

AI匹配打分的规则来源于笼中仙岗位画像文档（已获取，共10份）。以下为调研确认的核心维度框架：

| 维度 | 筛选逻辑 |
|------|---------|
| 行业匹配 | 优先餐饮行业经验；可接受酒店/服务业；完全不相关行业（销售、质检等）标记"不建议" |
| 年龄 | 按各岗位画像的偏好范围评分 |
| 稳定性 | 频繁跳槽（如3个月换一家）标记高风险 |
| 岗位特殊要求 | 如面点师必须能早起（关键筛选点） |
| 红线规则 | 一年内离职3次 → 直接过滤 |
| 基层特点 | 不强求简历完美，重实操能力相关信号 |

**注：** 具体筛选规则的字段级定义，在进入开发阶段前，需基于10份岗位画像文档完成规则提取。此处仅定义维度框架。

### 4.4 Boss直聘账号体系约束

- 近100家门店，每家用独立营业执照注册独立Boss账号
- 集团账号和单店账号两种类型并存
- 各门店账号在Boss直聘侧完全独立，互不关联
- V1需支持多账号管理：一个店长对应一个Boss账号

## 5. 不做清单

| 不做的事 | 原因 |
|---------|------|
| AI与候选人的多轮自动对话 | 蓝图明确排除；需先验证匹配判断准确率 |
| 沟通模式切换（快速筛选/重点跟进/文化传递/专业面试） | 属于体验优化层，V1先验证核心价值 |
| 线下面试辅助 | Phase 2范围 |
| 反馈闭环 | Phase 3范围 |
| JD撰写/职位发布 | 蓝图产品边界之外；笼中仙已有流程 |
| 薪酬谈判/Offer管理 | 蓝图产品边界之外 |
| 管理岗招聘（店长/管培生） | 蓝图产品边界之外；流程不同，需HR参与 |
| 非Boss直聘渠道（内推、老员工复职） | 蓝图当前阶段不覆盖 |
| 店长/HR模式切换 | V1只服务店长，HR为观察者 |
| 数据看板/管理后台 | V1先验证核心使用链路，后台能力后续补充 |

## 6. 验收标准

> Phase 1 的通过判断以核心交付验收和效果验收为主，扩展交付作为加分项单独评估。

### 6.1 核心交付验收（上线前必须通过）

| 验收项 | 通过条件 |
|--------|---------|
| 简历获取 | 系统能稳定读取指定Boss直聘账号的新简历信息，成功率达到可用水平（具体数值在技术验证阶段量化） |
| AI匹配打分 | 每份简历输出三档评分 + 风险标注，处理速度满足店长日常使用节奏（具体性能指标在技术验证阶段量化） |
| 筛选报告推送 | 店长在微信/飞书上收到报告，内容完整、可正常查看、关键信息一目了然 |
| 首轮接触自动化 | 店长确认后，系统成功对选定候选人完成首轮招呼动作，成功率达到可用水平（具体数值在技术验证阶段量化） |
| 多账号支持 | 支持试用范围内的门店账号同时运行，账号间数据隔离 |

### 6.2 扩展交付验收（若纳入当期迭代）

| 验收项 | 通过条件 |
|--------|---------|
| 输入法辅助 | 在指定场景下可基于候选人简历和对话上下文生成回复建议，店长可选择采纳或忽略 |
| 沟通汇总分析 | 在获取到聊天记录的前提下，能输出结构化沟通摘要（关键信息、风险点、亮点、是否建议约面试） |

> 扩展交付不作为 Phase 1 是否通过阶段验收的前提，是否纳入当期开发由技术验证结果与资源评估共同决定。

### 6.3 效果验收（上线后2-4周观察）

参照蓝图V2.2第八章成功标准，结合试用范围调整：

| 指标 | 目标 | 衡量方式 | 样本要求 |
|------|------|---------|---------|
| 店长筛简历时间 | 减少70% | 对比使用前后，店长每天花在Boss直聘上筛简历的时间 | 至少3家门店、连续观察2周 |
| AI推荐准确率 | ≥80% | AI标记"推荐沟通"的候选人中，店长实际愿意聊的比例 | 至少50份候选人样本 |
| 到面率 | 提升20% | 对比使用前后，邀约后实际到店面试的比例 | 至少3家门店、连续观察2周 |
| 店长续用意愿 | 愿意继续用 | 试用2周后，店长是否主动要求继续使用 | 所有试用门店店长 |

### 6.4 试用范围

- **门店数量：** 3-10家，优先选择招聘需求活跃的门店
- **岗位类型：** 至少覆盖2个岗位类型（如面点师 + 收银）
- **观察窗口：** 至少连续2-4周

## 7. 关键约束与依赖

| 项目 | 说明 |
|------|------|
| 岗位画像文档 | 已获取10份（doc/ppt/pdf/excel），开发前需完成规则提取 |
| Boss直聘技术可行性 | 简历获取方式、自动打招呼方式、登录保持机制——均在技术验证阶段（操作指南阶段B）确认 |
| 推送通道 | 微信还是飞书（或两者都支持），需确认笼中仙店长的实际使用情况 |
| 数据安全 | 数据库所有权归笼中仙；大模型API调用时内部数据不可被反向训练；候选人信息受《个人信息保护法》约束；笼中仙IT负责人需参与技术架构评审 |
| AI是副驾 | 业务状态变更权始终在店长手里。AI给出"不建议"不代表自动淘汰候选人。 |
| 入口形态与实现方式待定 | 核心交付的能力目标已确定（筛选、推送、首轮接触），但具体的产品入口形态和技术实现方式在技术验证阶段确认 |
